# 『Forza Horizon 6』スマートローラー連動システム 基本仕様書

## 1. システム概要

本システムは、PC上で動作する中継アプリケーション（C# / .NET）です。スマートローラー（FTMS対応デバイス）と『Forza Horizon 6』間で双方向のデータ通信を行い、自転車のペダリングによる車両の操作と、ゲーム内の地形変化に伴う自動負荷制御（自動連動）を実現します。

- **動作環境:** Windows 10 / 11 (Bluetooth 4.0以上搭載PC)
- **開発言語・フレームワーク:** C# / .NET 8 (または最新LTS)
- **対象ゲーム:** Forza Horizon 6 (PC版)

## 2. システムアーキテクチャ（構成）

将来的なデバイスの追加やロジックの変更に耐えうるよう、各モジュールを疎結合に設計します。例えば、FTMS通信部や入力エミュレーション部には、FactoryパターンやStrategyパターンを用いてインターフェースを切り出し、CYCPLUS T2のような特定機種固有の挙動や、将来的な他プロトコルへの拡張にも対応しやすい構造とします。

## 3. 機能仕様

### 3.1 入力変換（Power → Throttle）

スマートローラーから取得した出力（W）を、車両のアクセル開度（0.0〜1.0）に変換し、仮想コントローラーへ送信します。

- **FTPベースの正規化:** ユーザーの基準出力（FTPなど）を設定値として保持し、「設定値のX%」を「アクセル開度100%」としてマッピングします。
- **無入力時の処理（ゼロリセット）:** 一定時間（例: 1.5秒）パワーデータの更新がない、または0Wになった場合、即座にアクセル開度を0.0に戻します。

### 3.2 出力変換（Game Pitch → Trainer Resistance）

ForzaのUDPパケットから得られるピッチ角（車両の前後の傾き）を抽出し、斜度（%）に変換してスマートローラーの負荷を制御します。

- **ノイズフィルタリング:** ForzaのPitchデータは、サスペンションのバンプやジャンプ時の姿勢変化（高周波ノイズ）を含みます。これらが足への急激な負荷スパイクにならないよう、**移動平均（SMA）または指数移動平均（EMA）を用いたローパスフィルタ**を適用し、純粋な路面傾斜に近い値のみを抽出します。
- **負荷反映の遅延処理（スムージング）:** ゲーム内の地形変化（時速100km以上の車の挙動）と、自転車（時速20km）の体感ギャップを埋めるため、傾斜コマンドの送信頻度や変化率（スルーレート）に制限を設けます。

## 5. データ処理・アルゴリズム要件

### 5.1 傾斜（Pitch）から斜度（%）への変換

Forzaの Pitch 角度（ラジアン）を斜度（パーセンテージ）に変換する基本式は以下の通りです。

$$  \text{Grade (\%)} = \tan(\text{Pitch}) \times 100  $$

※上りがプラス、下りがマイナスになるよう、ゲームの座標系に合わせて符号を調整します。

### 5.2 速度差の補正（オプション要件）

「自転車のペダルを全力で回しても、Forza内の車が本来出す速度（時速200kmなど）に達してしまうと違和感がある」という問題への対策として、Forzaのゲーム内設定で対象車両のギア比を極端にローギヤードに設定するか、車重を重くするなどの「チューニング」を前提条件とします（プログラム側ではなくゲーム側での物理調整）。

UIフレームワーク等は、まずはコンソールアプリケーションとしてバックグラウンドで動く最小構成（MVP）を作り、ロジックが安定した段階でWPFやMAUI等で設定画面を被せるアプローチが効率的です。

## 追加仕様：デュアルマッピング・アーケード設計

### モードA: アーケードモード（Direct Throttle Strategy）

ペダルを踏む力（W）を、そのまま**車のアクセル開度（0.0〜1.0）に直結**させる、爽快感重視のモードです。

- **ロジック:** ユーザーのFTP（例: 200W）＝ アクセル100% としてマッピング。
- **特徴:** 200Wで漕ぐだけで車は時速200km以上まで加速します。現実の空気抵抗や自転車の物理法則を無視し、純粋に「漕いだ分だけスーパーカーがかっ飛ばす」というForza本来のゲーム性を楽しむためのモードです。

### モードB: シミュレーションモード（Target Speed Strategy）

Forza内の車を「自転車と同じ速度」に強制的に調律し、仮想サイクリング環境として機能させる、リアル志向のモードです。

- **ロジック (フィードバック制御):**

  1. スマートローラーのパワー(W)と、Forzaからの路面傾斜(%)から、「現実の自転車であれば現在何km/h出るか（目標速度）」を計算します。
  2. Forzaから取得している現在の車の速度（VelocityZ）と比較します。
  3. 目標速度に一致するように、プログラム側で自動的に仮想アクセルと仮想ブレーキを小刻みに調整（PID制御などを活用）します。
- **特徴:** どんなスーパーカーに乗っていても、平地で200Wなら時速30km程度しか出ず、下り坂では重力で勝手に加速します。CYCPLUS T2のような反応の良いスマートローラーによる傾斜負荷の自動変化と相まって、極めて没入感の高いバーチャルライドが実現します。

## Gemini の回答

シミュレーションモード（モードB）の設計において、「自転車の目標速度をどう割り出すか」と「Forzaの車をどうやってその速度に従わせるか」という2つのステップに分けて解説します。

自転車の速度は、単純な掛け算ではなく「空気抵抗」や「重力」といった複数の物理抵抗との釣り合いで決まるため、本格的な物理モデルの計算式を組み込む必要があります。

## 1. 自転車の物理モデル（目標速度の算出式）

スマートローラーが計測したパワー（ $P$ ）は、走行中に発生する「空気抵抗」「転がり抵抗」「勾配抵抗（重力）」の3つの抵抗を打ち消すために使われます。
これを方程式に表すと以下のようになります。

$$  P \cdot \eta = v \cdot \left( m \cdot g \cdot C_{rr} \cdot \cos(\theta) + m \cdot g \cdot \sin(\theta) \right) + \frac{1}{2} \cdot \rho \cdot C_d A \cdot v^3  $$

| 変数 | 意味 | 推奨される定数（ロードバイク想定） |
| --- | --- | --- |
| P | ローラーからの入力パワー | (FTMSから動的に取得) |
| v | 目標速度 (m/s) | これを計算して求める |
| θ | 道路の傾斜角 (rad) | (ForzaのPitchから動的に取得) |
| η | ドライブトレイン効率 | 0.97 (チェーンの伝達ロス) |
| m | 総重量（人 ＋ 自転車） | 75.0 kg |
| g | 重力加速度 | 9.81 m/s2 |
| Crr​ | 転がり抵抗係数 | 0.004 (アスファルト想定) |
| ρ | 空気密度 | 1.225 kg/m3 |
| Cd​A | 空気抵抗係数 × 前面投影面積 | 0.32 m2 (ブラケットポジション) |

この式を見るとわかる通り、目標速度  $v$  は**3次方程式**になっています。これを毎フレーム解析的に解くのは少し手間なので、C#のプログラム上ではニュートン・ラフソン法（近似解法）を用いて、前フレームの速度を初期値として高速に収束させるのが最もスマートな実装です。

以下は、各要素が速度にどう影響するかを視覚化した計算ウィジェットです。

## 2. PID制御の設計方針（車両の速度追従）

目標速度（ $V_{target}$ ）が求まったら、次はForzaの車をその速度にピッタリ合わせます。ここで登場するのが**PID制御**です。

Forzaから受信した車両の現在速度（ $V_{car}$ ）と目標速度の「誤差」を計算し、それをもとにアクセル（Throttle）とブレーキ（Brake）の踏み込み量を決定します。

### 制御アルゴリズムの基礎

1. **誤差の算出 ( $e$ )**

$$  e(t) = V_{target} - V_{car}  $$
2. **PID出力の計算 ( $U$ )**

$$  U(t) = K_p \cdot e(t) + K_i \int e(t) dt + K_d \frac{de(t)}{dt}  $$

（※  $K_p, K_i, K_d$  は調整用のゲイン・パラメータ）
3. **コントローラー入力へのマッピング**

  - $U(t) > 0$  の場合： Throttle  $= \min(U(t), 1.0)$ 、Brake  $= 0.0$
  - $U(t) < 0$  の場合： Throttle  $= 0.0$ 、Brake  $= \min(-U(t), 1.0)$

### C#での実装イメージとアンチワインドアップ対策

ゲーム連動特有の課題として、「壁に激突して車が動けないのに、プレイヤーがペダルを全力で漕ぎ続けている状況」**があります。
このとき、誤差（目標速度 － 現在速度ゼロ）が蓄積し続け、積分項（ $I$ ）が無限に巨大化してしまいます（ワインドアップ現象）。壁から離脱した瞬間に車が暴走してしまうため、積分項の上限と下限を切り詰める**アンチワインドアップ（積分飽和対策）が必須です。

C#

```
public class SpeedPidController
{
    private float _kp = 0.5f;
    private float _ki = 0.1f;
    private float _kd = 0.05f;

    private float _integral = 0f;
    private float _previousError = 0f;
    private float _maxIntegral = 5.0f; // アンチワインドアップ用の上限値

    public ControlOutput Compute(float targetSpeed, float currentCarSpeed, float deltaTime)
    {
        float error = targetSpeed - currentCarSpeed;

        // P (比例)
        float p = _kp * error;

        // I (積分) と アンチワインドアップ制限
        _integral += error * deltaTime;
        _integral = Math.Clamp(_integral, -_maxIntegral, _maxIntegral);
        float i = _ki * _integral;

        // D (微分)
        float derivative = (error - _previousError) / deltaTime;
        float d = _kd * derivative;

        float output = p + i + d;
        _previousError = error;

        // 出力をアクセル(0~1)とブレーキ(0~1)に振り分け
        return new ControlOutput
        {
            Throttle = Math.Clamp(output, 0f, 1f),
            Brake = output < 0 ? Math.Clamp(-output, 0f, 1f) : 0f
        };
    }
}

```

### チューニングのコツ

- ** $K_p$  (比例ゲイン):** 最初はこれだけを設定します。値を大きくすると目標速度へ素早く到達しようとしますが、大きすぎると車が前後にガクガク揺れるハンチング現象が起きます。
- ** $K_i$  (積分ゲイン):** 坂道などで「あともう少しで目標速度に届かない」という定常偏差をジワジワと埋める役割です。
- ** $K_d$  (微分ゲイン):** 急に速度が変わったときの「ブレーキの掛け遅れ」や「アクセルの踏みすぎ」を抑え、挙動をなめらかにします。

この設計であれば、下り坂で  $V_{target}$ （目標速度）が重力によって  $60km/h$  に跳ね上がった際、車側がまだ  $40km/h$  ならPIDがフルスロットルを指示し、逆にプレイヤーが足を止めて  $V_{target}$  が  $0km/h$  になれば、PIDが即座にマイナス方向（ブレーキ）を指示するため、非常に自然な自転車の挙動を再現できます。

## 6. コンソールログ出力の解説

本ブリッジアプリケーションの実行中、コマンドプロンプトやPowerShell上にリアルタイムに表示されるログの各パラメータおよび値の意味を以下に解説します。

### 6.1 `[ACTIVE]` 動作状況のリアルタイム表示行
走行中、最新のステータス情報が1行でリアルタイム更新（上書き表示）されます。

#### ■ シミュレーションモード（Simulation Mode）実行時
```text
[ACTIVE] SIMULATION MODE | Pedal: 140 W | Target: 29.8 km/h | Car: 2.9 km/h | Grade: 1.6% (Diff: 50%) | Out -> Thr: 0.98, Brk: 0.00
```
*   **`SIMULATION MODE`**: 現在選択されている動作モードが「シミュレーションモード」であることを示します。
*   **`Pedal`**: スマートローラーからBluetooth LE経由で受信した、ユーザーの現在の瞬時ペダリングパワー（W / ワット）。
*   **`Target`**: 自転車物理モデルに基づき、現在のパワーと道路傾斜から「本来自転車であれば出るはずの目標速度」（km/h）。
*   **`Car`**: Forzaゲーム内から受信した、現在の車の前進方向のリアルタイム速度（km/h）。
*   **`Grade`**: サスペンション補正、難易度（Difficulty）調整、および仮想ギアダウン補正がすべて適用された後の、最終的なスマートローラー制御用斜度（%）。
*   **`(Diff: XX%)`**: ユーザーが設定したスマートローラーの「負荷再現割合（難易度）」。
*   **`Out -> Thr`**: 車速を目標速度に追従させるため、PID制御が計算した「アクセル（スロットル）開度」（`0.00` 〜 `1.00` = 0%〜100%）。
*   **`Brk`**: PID制御が計算した「ブレーキ開度」（`0.00` 〜 `1.00`）。※ペダリング中は自動操舵アシスト維持のため強制的に `0.00` にクリアされます。

#### ■ アーケードモード（Arcade Mode）実行時
```text
[ACTIVE] ARCADE MODE | Pedal: 104 W | Direct Accel: 52% | Grade: -0.3% (Diff: 40%) | Out -> Thr: 0.52, Brk: 0.00
```
*   **`ARCADE MODE`**: 現在選択されている動作モードが「アーケードモード」であることを示します。
*   **`Direct Accel`**: 現在のペダリングパワーを基準FTP値で割った「ダイレクトアクセル開度（%）」。この値がそのまま車のアクセル入力になります。

### 6.2 `[DEBUG-TELEMETRY]` ゲーム内テレメトリ受信行
ForzaからUDP経由で受信した、車の生のテレメトリ情報が1秒に1回表示されます。
```text
[DEBUG-TELEMETRY] Time: 30058187 | RawPitch: -0.1834 rad | Accel: X:0.01, Y:-0.03, Z:0.01 | Speed: 2.9 km/h
```
*   **`Time`**: Forzaのゲーム内タイムスタンプ（ミリ秒）。
*   **`RawPitch`**: Forzaから受信した車体の生ピッチ角（ラジアン）。※Forzaの座標系では「車首が上がると負（マイナス）の値」を取るため、符号反転して勾配計算に使用されます。
*   **`Accel X, Y, Z`**: ゲーム内の車体が受けている3軸の加速度（G）。
    *   **`X`**: 左右方向の加速度。
    *   **`Y`**: 上下方向の加速度。
    *   **`Z`**: 前後（進行）方向の加速度。この前後加速度に比例した係数を掛けることで、急加減速時のサスペンション姿勢沈み込みノイズを相殺します。
*   **`Speed`**: ゲーム内の車の現在速度（km/h）。

*   **`Time`**: コマンド送信時のゲーム内タイムスタンプ（ミリ秒）。
*   **`Filtered`**: 姿勢変化や急な変化をなめらかにするため、EMA（指数移動平均）フィルターおよびスルーレート制限を適用した後の補正後斜度（%）。
*   **`SentToTrainer`**: 実際にスマートローラーへ送信されたコマンド情報。
    *   **`FREE (OpCode 0x04, Level 0) [Descent/Flat]`**: 斜度が平地または下り坂（`targetIncline <= 0%`）の時、スマートローラー内部の速度依存空気抵抗を完全に排除するため、負荷レベルを強制的に「0 (完全スピンフリー)」として送信したことを示します。
    *   **`X.X% (Trigger: Normal)`**: 上り坂（`targetIncline > 0%`）の時、`X.X%` の斜度シミュレーションパラメータ（FTMSプロトコル OpCode 0x03）を送信したことを示します。
